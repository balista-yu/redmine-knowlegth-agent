# コードレビュー

直近の変更 (デフォルト: HEAD と作業ツリーの差分) をレビューする。引数として追加コンテキストやレビュー対象パスを受け取れる。

## レビュー観点 (優先度順)

1. **アーキテクチャ違反**: `.claude/rules/architecture.md` と該当層の rules ファイルに照らして違反がないか
   - Domain 層に Spring/Koog/JPA 等の import が混入していないか
   - Application が Infrastructure を import していないか
   - interface の戻り値・引数に SDK 型が露出していないか
2. **責務分離**: Domain Service が I/O を持っていないか、Application が直接 SDK を叩いていないか、Controller が Repository を直接呼んでいないか
3. **テスト網羅性**: 該当タスクの「テスト要件」が満たされているか、エッジケースが漏れていないか
4. **テストの質**: 偽陽性・偽陰性を生みやすい assertion になっていないか、実装の鏡写しテストになっていないか
5. **Clean Code**: マジック値、長すぎる関数 (30 行超)、`!!`、意図不明の命名
6. **命名・配置**: `docs/01-design.md` §6.4 のディレクトリ構成・命名規約と一致しているか
7. **API 契約**: web 層の場合、`docs/03-api-spec.md` と一致しているか
8. **エラーハンドリング**: `docs/01-design.md` §8.5 の方針 (リトライ・例外変換) に従っているか

## 進め方

1. `git status` と `git diff HEAD` で変更内容を把握
2. 必要に応じて変更ファイルを `Read` で確認 (周辺コードも文脈把握のため)
3. 各観点について確認
4. レポート出力

## 出力形式

```markdown
## レビュー結果

### サマリ
- 総合判定: PASS / FAIL / CONDITIONAL_PASS
- 重大な問題: N 件
- 軽微な指摘: M 件

### 重大な問題 (FAIL を構成)
1. [観点] <ファイル>:<行>
   - 問題: <何が問題か>
   - 修正案: <具体的にどう直すか>

### 軽微な指摘
1. [観点] <ファイル>:<行>
   - 問題: ...
   - 修正案: ...

### 良かった点
- ...

### 追加で提案する ArchUnit ルール (あれば)
- ...
```

## 制約

- 推測ではなく **実際のコードに基づいて** 指摘する
- 既存設計書と矛盾する指摘は避ける (設計書への異議があれば「設計書見直しが必要」と明記)
- 「LGTM」だけで終わらせない (1 つ以上の良かった点 or 改善提案を出す)
