# 設計判断の相談

引数で渡された相談内容について、複数案を比較して推奨を提示する。

## 進め方

1. `docs/01-design.md` および関連する `.claude/rules/*.md` を確認
2. 必要なら関連コード (既存実装) を `Read`
3. 2〜3 案を比較
4. 推奨案と理由を提示

## 出力形式

```markdown
## 設計判断: <タイトル>

### 制約条件 (確認)
- スケール: 同時 1〜数人、チケット 〜数千件
- 認証なし (個人/小チーム検証)
- オニオンアーキテクチャ遵守
- <その他、相談固有の制約>

### 案 1: <名称>
- 概要: 1〜2 行
- メリット:
  - ...
- デメリット:
  - ...
- アーキテクチャ純度: ★/★★/★★★ (オニオンの原則からの逸脱度)
- 実装コスト目安: S/M/L
- テスト容易性: ★/★★/★★★
- 推奨度: ★/★★/★★★

### 案 2: <名称>
...

### 案 3: <名称>
...

### 推奨

<どれを推すか + 理由 (2〜3 行)>

### 推奨案を採用する場合の影響範囲
- `docs/01-design.md` の更新が必要か: <はい / いいえ>
- 影響するタスク: <T-X-Y, T-X-Z 等>
- 既存コードの変更: <あり / なし>
```

## 厳守

- **コードは書かない**: 設計検討のみ
- 推測ではなく **根拠ベース** で比較する (Koog のドキュメント・既存コード・設計書を参照)
- 「全部メリットあります」ではなく **トレードオフを明確に** する
- 推奨案は 1 つに絞る (「ケースバイケース」で逃げない)
